## Workflow

### 1. Choose Scope

Default command:

```bash
cargo test
```

For a narrow code change, first run the smallest test command that covers the
changed behavior if the repo supports targeted tests. Then run the configured
command when the task, repo rules, or risk level requires the full test gate.

Do not invent a test command. Derive targeted commands from existing repo
scripts, test framework conventions, or nearby test files.

### 2. Run Tests

Record:

- exact command
- pass/fail count where available
- failing test names
- failure file paths and line numbers
- relevant stderr/stdout

### 3. Classify Failures

| Class                  | Signals                                    | First checks                              |
| ---------------------- | ------------------------------------------ | ----------------------------------------- |
| Product regression     | expected/actual mismatch in behavior       | Read source and test together             |
| Test expectation drift | product behavior intentionally changed     | Verify with user request and repo docs    |
| Fixture/mock issue     | missing setup, stale mock, wrong seed data | Inspect test setup before production code |
| Async/timing issue     | timeout, race, unawaited promise           | Trace lifecycle and cleanup               |
| Environment issue      | missing binary, permission, network/auth   | Report blocker before changing app code   |

### 4. Fix Or Explain

When the user asked for fixes:

1. Read the failing test and implementation before editing.
2. Fix the root cause rather than weakening assertions.
3. Add or update regression coverage when behavior changed.
4. If the evidence points to local setup, permissions, missing services, or
   machine-specific tooling, report the setup fix before changing application
   code.
5. Re-run the targeted test, then the configured command when appropriate.

When the user only asked to run or analyze tests, do not edit code without
asking.

### 5. Report

Report exact commands and results. If tests still fail, list remaining failures
and the likely next step.

## Repo Overlay

### gitterm-v2 Test Notes

#### Feature Coverage

`cargo test` covers the default feature set (~96 tests). `cargo test --features
excalidraw` adds the diagram-viewer tests (~108 total). CI runs both — run both
locally for parity:

```bash
cargo test
cargo test --features excalidraw
```

#### Test Canaries

- **`tests::collect_file_tree_hides_dotfiles`** (`src/main.rs:14184`) is the
  canary for `services::collect_file_tree` correctly honoring `show_hidden`.
  Pre-TRU-69 this test was failing because the parameter was accepted but
  ignored. If it ever fails again, suspect the same root cause first.
- **`plans_viewer::tests::*`** cover path-traversal safety on
  `/plans/raw/{name}`. Don't regress these without scrutiny — they protect
  against directory escapes and non-`.md` requests.

#### Setup Requirements

No external services required. Tests use `tempfile` for filesystem fixtures
and don't touch the network or the warp log server. The test binary doesn't
launch the Iced GUI.

#### Test Layout

Most tests live inline in `src/main.rs` under `#[cfg(test)] mod tests`. The
plans-viewer tests live in `src/plans_viewer.rs::tests`. There's no separate
`tests/` integration directory.
