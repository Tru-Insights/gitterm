## Command

```bash

```

## Workflow

### 1. Confirm The Command

Use the configured command as the source of truth. If the command is missing,
not available, or contradicted by repo instructions, stop and call out the
conflict before guessing.

### 2. Run Build

```bash
cargo build
```

Capture the exact failing output. Do not summarize away file paths, line
numbers, stack frames, missing modules, or bundler/runtime names.

### 3. Classify Failures

Look for the actual failure class before editing:

| Class                  | Common signals                                                    | First checks                                                                    |
| ---------------------- | ----------------------------------------------------------------- | ------------------------------------------------------------------------------- |
| Type errors            | incompatible value, missing property, bad import type             | Read the source file and the type definition                                    |
| Module/import errors   | cannot resolve module, missing export, bad path                   | Check import path, export shape, config/import map                              |
| Build/runtime boundary | SSR/client mismatch, bundled node built-in, env access            | Check repo architecture rules and bundler config                                |
| Asset/config errors    | missing file, raw import failure, env var missing                 | Check checked-in config and documented env requirements                         |
| Local setup/toolchain  | missing binary, wrong runtime version, cache/package layout issue | Check versions, lockfiles, install steps, and CI behavior before editing source |
| Syntax/format errors   | parser failure, malformed generated file                          | Open the exact file and nearby lines                                            |

### 4. Fix Narrowly

For each failure:

1. Read the failing file and the nearby call sites.
2. Verify expected contracts from existing code, docs, or config.
3. If the evidence points to local setup, report the setup fix before changing
   application code.
4. Apply the smallest fix that preserves intended behavior.
5. Avoid broad refactors, dependency changes, or fallback logic unless the user
   explicitly asked for that scope.

### 5. Re-Run And Report

Re-run:

```bash
cargo build
```

Report:

- final status
- exact command run
- failures fixed, with file paths
- any warnings or residual risks
- whether follow-up tests/checks are still needed

Do not substitute a package-manager command that is not configured for this
repo.

## Repo Overlay

### gitterm-v2 Build Pipeline

Full reference is `BUILD.md` at the repo root — Linux deps, cross-compilation,
GitHub Actions, etc. The configured `cargo build` is just the dev-loop default;
shipping has more flags and a bundle step.

#### Common Commands

| Need                  | Command                                           |
|-----------------------|---------------------------------------------------|
| Dev loop              | `cargo run`                                       |
| Quick debug build     | `cargo build`                                     |
| Release binary        | `cargo build --release --features stt`            |
| Ship macOS .app       | `./scripts/bundle.sh`                             |
| Install bundle        | `cp -r target/GitTerm.app /Applications/`         |
| Launch installed app  | `open /Applications/GitTerm.app`                  |

#### Feature Flags

- `stt` — voice (whisper-rs + cpal). Required for shipping. Off by default.
- `excalidraw` — diagram viewer surface. Optional. Off by default.

`./scripts/bundle.sh` builds with both (`stt excalidraw`).

#### Before triggering GitHub Actions CI

All three workflows (`build.yml`, `ci.yml`, `windows-build.yml`) clone
`Tru-Insights/iced_term` master fresh on every run. If you've made local
commits to `../iced_term_fork` that gitterm depends on, push them first or
CI fails with `no method named X found` errors:

```bash
cd ../iced_term_fork
git log @{u}..HEAD --oneline   # must be empty before CI
git push origin master          # if it's not
```

#### Output Locations

- `cargo build` → `target/debug/gitterm`
- `cargo build --release ...` → `target/release/gitterm`
- `./scripts/bundle.sh` → `target/GitTerm.app` (custom bundle, multi-instance launcher)
- `cargo bundle --release --features stt` → `target/release/bundle/osx/GitTerm.app`
  (cargo-bundle crate output; use `bundle.sh` instead for the multi-instance launcher)
