# PR Ready

Generated by `agent-workflow-kit`. Edit the kit template,
`agent-workflow.config.json`, or a repo overlay, not this rendered file.

Read `AGENTS.md` before deciding readiness. Repo instructions override this
generic workflow when they are stricter.

## Config

- Project: `gitterm-v2`
- Default base branch: `master`
- Protected branches: `master`
- Issue provider: `linear`
- Issue key pattern: `[A-Z][A-Z0-9]{1,9}-[0-9]{1,7}`
- Issue required in PR: `true`
- GitHub readiness attestation required: `true`
- PR template path: `.github/pull_request_template.md`
- Verification before PR-ready: `check, test, build`
- Manual verification notes allowed: `true`

## Mode

Choose the mode from the user's target.

| Target                          | Mode           | Meaning                                       |
| ------------------------------- | -------------- | --------------------------------------------- |
| No PR number or URL             | Current branch | Author preflight before opening/requesting PR |
| `--pr <number>` or a PR URL     | Explicit PR    | Final readiness gate for an existing PR       |
| Review/approve/merge an open PR | Explicit PR    | Never use the current branch by accident      |

Current-branch mode can say the branch is ready to open as a draft or ready to
run explicit PR mode later. It is not proof that an existing PR is ready to
merge.

Explicit PR mode is required before requesting review, marking ready, approving,
or merging when `github.requirePrReadyBeforeReview` is true.

## Workflow

### 1. Load Repo Rules

Read `AGENTS.md`, then scan for stricter PR, testing, issue, security,
architecture, or ownership rules. If the repo provides a local context
collector, PR checklist, or validation script, use it before falling back to
manual Git/GitHub commands.

Do not rely on a PR summary, intent, or manual notes alone.

### 2. Collect Target Context

For current-branch mode:

```bash
git status --short --branch
git merge-base HEAD origin/master
git diff --name-status origin/master...HEAD
git log --oneline origin/master..HEAD
```

Adjust the base branch if repo instructions, branch naming, or the PR target
prove a different lane is intended.

For explicit PR mode, fetch the PR metadata and actual diff. Prefer `gh` or the
available GitHub connector for:

- title, body, author, base branch, head branch, head SHA
- changed files and patch
- commits
- requested reviewers and draft/ready state
- CI/check status for the current head SHA
- prior review submissions and unresolved review threads
- linked issues or issue keys in title/body/commits/branch

If the PR target cannot be identified, stop and ask for the PR number or URL.

### 3. Check Hard Blockers First

Block readiness for any of these unless repo instructions explicitly say the
case is acceptable:

- Missing required issue key matching `[A-Z][A-Z0-9]{1,9}-[0-9]{1,7}` when issue
  linkage is required by config or repo rules.
- Scope is not atomic: unrelated tickets, cleanup, refactors, dependency bumps,
  or generated churn are bundled with behavior work.
- Base branch does not match the intended lane.
- Prior requested changes or unresolved threads are not explicitly addressed.
- Behavior changed without targeted automated regression coverage, unless the
  repo permits manual-only verification and the limitation is stated.
- Required verification commands are failing, skipped, or only assumed.
- CI is failing, pending too long to judge, or attached to a stale head SHA.
- Submodule, generated, lockfile, dependency, runtime, or build config changes
  are unexplained.
- Repo ownership boundaries are crossed without the documented upstream change.
- Required data contracts, endpoint paths, field names, status values, enum
  values, or props are guessed instead of verified.
- Error handling swallows failures or loses useful operation/input context.
- Build/runtime risks are unverified after touching bundlers, SSR/client
  boundaries, dynamic imports, environment variables, assets, or production-only
  paths.
- Security basics regress: secrets, unsafe logging, XSS/injection risk,
  overbroad permissions, auth bypass, missing authorization boundary, or
  sensitive data exposure.
- The repo requires a current-head readiness attestation and the PR body lacks
  one for the current head SHA.

Add repo-specific blockers from `AGENTS.md` and treat them as equally
authoritative.

### 4. Review The Diff

Read every changed file that matters to behavior. For each change, verify:

- the data flow and state transitions are correct
- edge cases and stale/race conditions are handled
- tests cover the behavior that changed
- external contracts match existing code, docs, or backend definitions
- generated files are intentional and reproducible
- unrelated changes are removed or split out

Use exact file and line evidence for each blocker. Avoid style-only nits unless
they hide a correctness, maintainability, or repo-rule risk.

### 5. Run Verification

Run the configured verification commands that exist for this repo. The default
PR-ready set is:

```text
check, test, build
```

Use the configured command strings:

```bash
cargo fmt -- --check && cargo clippy -- -D warnings
cargo test
cargo build
```

Also run `lint`, `build`, targeted tests, browser checks, or migration checks
when repo instructions, touched files, or `verification.beforePrReady` require
them. If a configured verification command is not available, stop and call out
the config/repo mismatch instead of substituting an unrelated package-manager
default.

### 6. Record Explicit PR Attestation

Only do this in explicit PR mode, only after all blockers are cleared, and only
for the current PR head SHA.

If `github.requirePrReadyBeforeReview` is true, update the PR body so it
contains a current-head block:

```markdown
## Review readiness

- [x] `/pr-ready --pr <number>` returned `ACCEPT-READY`
- Head SHA: <current 40-character PR head SHA>
- Reviewed at: <ISO-8601 UTC timestamp>
```

If the PR already has this section, replace it instead of adding a duplicate. If
any new commit lands after the attestation, the attestation is stale and
explicit PR mode must be rerun.

If you cannot update the PR body and the repo requires the attestation, return
`BLOCKED`, not `ACCEPT-READY`.

## Output

Lead with the status:

- `ACCEPT-READY` when no blockers remain.
- `BLOCKED` when the PR or branch must not request review, be marked ready,
  approved, or merged.
- `NEEDS-INFO` when a required target, issue, contract, credential, or command
  cannot be determined.

Then report:

- mode and target
- base branch and head SHA, when available
- issue linkage result
- verification commands run and results
- concrete blockers ordered by severity, with file/line evidence
- residual risks or coverage gaps even when accepted
- whether the readiness attestation was written, stale, not required, or not
  applicable

Keep the final recommendation explicit. Do not request review, approve, mark
ready, or merge when status is not `ACCEPT-READY`.
